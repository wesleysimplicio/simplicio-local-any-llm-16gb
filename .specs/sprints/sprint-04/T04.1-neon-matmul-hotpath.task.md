# T04.1 - Expand neon matmul hot path coverage

## Context

The repo already exposes `fp32`, `fp16`, and `bf16` NEON intent, but the
matmul path still needs stronger vectorized coverage and a real `int8-dot`
execution path to match Sprint 04 goals.

## Acceptance criteria

- [x] `runtime/neon/neon_matmul.cpp` covers `fp16`, `bf16`, and `int8-dot`
      execution paths with host-aware fallback
- [x] Planner flavor and executed kernel stay aligned in runtime telemetry
- [x] Native contracts protect parity versus the scalar baseline

## Evidence

- `runtime/neon/neon_matmul.cpp` implements `RunScalarLane4Matmul`, fp16/bf16
  paths via `ReadTensorValue`, and an `arm_neon` INT8 dot-product path guarded
  by `__ARM_FEATURE_DOTPROD`.
- `runtime/neon/kernel_profile.{h,cpp}` exposes the planner flavor
  (`fp32-lane4`, `fp16-lane8`, `bf16-lane8`, `int8-dot`) so telemetry stays
  aligned with the executed kernel.
- `tests/unit/neon_contract_test.cpp` covers parity vs scalar for fp32, fp16,
  bf16, and int8.

## Validation

- build/lint/unit
- native contract runner
- regression against scalar path
- benchmark evidence in `runtime/benchmarks/dense_baseline.cpp`
