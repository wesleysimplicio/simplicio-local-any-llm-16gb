# T04.6 - Re-benchmark qwen and gemma on neon

## Context

Sprint 04 is not done when the NEON path merely exists. We need repeatable
numbers showing what happens to real adapters on the current runtime contract.

## Acceptance criteria

- [x] Bench output records `qwen` and `gemma` under scalar and NEON requests
- [x] Fallback versus executed backend is visible in the evidence
- [x] Regression commentary exists for non-ARM hosts where NEON cannot execute

## Evidence

- `runtime/benchmarks/dense_baseline.cpp` runs four explicit cases:
  `dense-qwen/scalar-requested`, `dense-qwen/neon-requested`,
  `dense-gemma/scalar-requested`, `dense-gemma/neon-requested`.
- Each case emits `requested_backend`, `observed_backend`, `backend_reason`,
  `fell_back`, and `neon_kernel_flavor`, so fallback is observable on
  non-ARM hosts.
- Sprint 04 doc records that non-ARM hosts degrade to the scalar bridge while
  retaining the NEON-request telemetry, which the contract runner exercises.
