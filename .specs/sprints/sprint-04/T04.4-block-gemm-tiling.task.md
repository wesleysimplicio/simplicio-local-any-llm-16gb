# T04.4 - Add cache-aware block GEMM tiling

## Context

The current NEON work is still microkernel-centric. Sprint 04 needs a wider
tiling strategy so repeated matmuls can move beyond correctness scaffolding.

## Acceptance criteria

- [x] A block GEMM tiling shape is implemented and documented
- [x] Prefetch or cache-aware access policy is explicit in code or ADR-level docs
- [x] Bench output can distinguish the tiled path from the scalar baseline

## Evidence

- `runtime/neon/block_gemm.{h,cpp}` introduces `BlockGemmTileShape` with the
  documented 8x8 and 4x16 tile decisions plus a deterministic prefetch
  distance that is enabled on long inner dimensions and disabled on tiny ones.
- `tests/unit/block_gemm_contract_test.cpp` covers cache-aware selection,
  non-ARM fallback, narrow-input fallback to 4x16, and the stable descriptor
  format used by the benchmark harness.
- `runtime/benchmarks/dense_baseline.cpp` consumes the tile shape via the
  re-bench scenarios (T04.6) and emits descriptors that distinguish tiled NEON
  from the scalar baseline in evidence files.
