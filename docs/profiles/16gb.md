# Perfil `16gb` (issue #119, epica #116)

## Objetivo

Um perfil de execucao nomeado (`--profile 16gb` na CLI `coli`, ou
`COLI_PROFILE=16gb` direto no binario `./glm`) que garante **por construcao**
RSS <= 13 GB, escolhendo defaults que maximizam qualidade/velocidade dentro
desse teto — sem exigir que o usuario configure cada variavel manualmente.

## Orcamento de memoria (por que 13 GB, nao 16 GB)

Numa maquina de 16 GB:

| item                          | orcamento |
|-------------------------------|----------:|
| SO + outros processos        |    ~3 GB  |
| tronco denso (residente)     | <= 8.5 GB |
| cache de experts (streaming) | 1.5-3 GB  |
| KV cache + buffers de runtime | <= 1 GB  |
| **teto de RSS do processo**  | **13 GB** |

O teto de 13 GB e' o que este perfil configura (`COLI_RSS_CEILING_GB=13`
e `RAM_GB=13`), deixando ~3 GB de folga para o SO.

## O que o perfil aplica (e por que)

Ativado com `COLI_PROFILE=16gb` (env) ou `coli ... --profile 16gb` (CLI).
Cada default so' e' aplicado se o usuario **nao** tiver setado a variavel
explicitamente (uma flag/env explicita sempre ganha do perfil).

| variavel                  | valor | razao |
|----------------------------|-------|-------|
| `MTP`                      | `0`   | A testa MTP (draft speculativo nativo) exige manter pesos extras residentes e, a frio (sem historico de acceptance quente), gera mais expert-loads adicionais do que economiza em token/forward sob uma cache de experts pequena — o cenario tipico de 16 GB. Ver `tools/convert_fp8_to_int4.py`: MTP a int4 tem acceptance ~0-4% medida (issue #8 do colibri upstream); mesmo a int8 (39-59%) o ganho so' compensa com cache de experts generosa, que 16 GB nao tem. |
| `DSA`                      | `1`   | O indexer DSA (lightning indexer) faz atencao ESPARSA acima de `index_topk` tokens de contexto, reduzindo compute e a necessidade de reconstruir K/V para todo o contexto — importante quando o KV buffer de runtime tem que caber no ~1 GB restante do orcamento. |
| `TEMP`                     | `0.7` | Temperatura de sampling moderada: reduz a chance de o modelo divagar para tokens de baixa probabilidade justamente na cauda que a quantizacao agressiva (int4/int2) torna mais ruidosa. |
| `TOPP`                     | `0.7` | Top-p adaptativo na SELECAO DE EXPERTS (nao no vocabulario — isso e' `NUCLEUS`): com cache pequena, restringir a selecao aos experts de maior peso cumulativo reduz cache-miss/disk-load sem descartar o expert que o roteador considera essencial. |
| `AUTOPIN`                  | `1`   | Pin learning: o motor acumula o historico de uso de experts em `<model>/.coli_usage` entre sessoes e fixa (pin) os mais usados em RAM, dado historico suficiente (>=5000 selecoes, ver `pin_load`/`usage_load` em `glm.c`). Sob orcamento de cache pequeno, aprender quais experts sao "quentes" para ESTA maquina/uso vale mais do que uma LRU pura. |
| `COLI_RSS_CEILING_GB`      | `13`  | Teto do gate duro de RSS (secao abaixo). |
| `RAM_GB`                   | `13`  | Budget que `cap_for_ram` usa para dimensionar a cache de experts dentro do mesmo teto. |

Implementacao: bloco no inicio de `main()` em `engine/c/glm.c` (antes de
qualquer outro `getenv`), e `--profile`/`COLI_PROFILE` no wrapper `coli`
(`engine/c/coli`, funcao `env_for`) que so' repassa a variavel de ambiente —
os defaults concretos vivem no motor C, entao funcionam tambem chamando
`./glm` direto, sem passar pela CLI Python.

```bash
# via CLI
coli run --profile 16gb --model /path/to/glm52_i4 "prompt"
coli plan --profile 16gb --model /path/to/glm52_i4   # rejeita o plano ANTES de iniciar o motor, se nao couber

# direto no binario
COLI_PROFILE=16gb SNAP=/path/to/glm52_i4 ./glm 64 4 4
```

## Gate duro de RSS

Alem dos defaults, o perfil configura um TETO que o motor VERIFICA antes de
rodar. Implementado em duas camadas:

1. **C (`engine/c/glm.c`, `rss_hard_gate`)**: apos `model_init` (pesos
   densos ja carregados, `resident_bytes` conhecido) e ANTES de alocar
   cache de experts/PIN, projeta o pico MINIMO possivel — tronco denso +
   UMA fatia de cache por camada sparse (`cap=1`, a menor cache que
   `cap_for_ram` jamais produziria) + slack honesto de KV/working-set/
   page-cache/overhead (mesma formula de `cap_for_ram`). Se esse minimo
   ja excede `COLI_RSS_CEILING_GB`, o processo **aborta de forma limpa**
   (`exit(3)`, mensagem acionavel em stderr) — nao tenta rodar e nao
   espera o OOM-killer do kernel matar o processo no meio da geracao.
2. **Python (`engine/c/resource_plan.py`, `check_rss_ceiling` +
   `project_minimum_peak_bytes`)**: a mesma projecao, aplicada ao PLANO
   (antes de qualquer processo do motor ser iniciado). `coli plan --profile
   16gb` chama isso e recusa um plano inviavel na hora, sem gastar tempo de
   carregamento.

`COLI_RSS_CEILING_GB<=0` (ou ausente) desativa o gate — comportamento
legado, idêntico ao motor antes desta issue, para quem nao usa o perfil.

Teste automatizado (sem precisar de hardware real de 16 GB — e' logica de
projecao, testavel com qualquer tamanho de modelo):

- `engine/c/tests/test_16gb_profile.py::test_rss_gate_aborts_cleanly_when_projection_exceeds_ceiling`
  — roda o binario `./glm` real contra a fixture do oraculo com um teto
  microscopico de proposito, e prova exit code positivo + mensagem `[RSS
  GATE] ABORT` em stderr, sem crash/signal.
- `engine/c/tests/test_resource_plan.py::RssHardGateTest` — testa
  `check_rss_ceiling` isoladamente (sem subprocess, sem modelo real).

## Quantizacao mista (container v2) — a ferramenta para caber no sub-orcamento de 8.5 GB

O motor, tal como vendorizado (issue #117), mede o tronco denso do GLM-5.2
em int4 em ~9.9 GB (medicao upstream, documentada no README do colibri) —
ACIMA do sub-orcamento de 8.5 GB deste perfil. O motor ja tem kernels int2
empacotados validados bit-identicos (`quant_int2`/`pack_int2` em
`tools/convert_fp8_to_int4.py` / `glm.c`) que, antes desta issue, nunca
eram usados para reduzir especificamente o residente.

O container v2 (`tools/convert_fp8_to_int4.py --dtype-map <mapa.json>`)
permite aplicar int2 SELETIVAMENTE a tensores especificos do tronco denso
(ex.: `embed_tokens`/`lm_head`, ou camadas de atencao especificas — ver
`docs/baselines/glm52-dense-breakdown.md` para quais categorias sao maiores
candidatas), mantendo o resto em int4/int8, para tentar encaixar o tronco
denso real dentro dos 8.5 GB sem sacrificar tudo uniformemente. O `mapa.json`
e' `{"regex-do-nome-do-tensor": bits}`; o conversor grava
`container_manifest.json` no outdir documentando o bits final por tensor, e
o loader C valida essa declaracao contra o tamanho real em disco (aborta
com mensagem clara se nao combinarem — protege contra corrupcao de
container).

**Isto NAO foi validado com o checkpoint GLM-5.2 real** (indisponivel neste
ambiente, ~370 GB). Foi testado com a fixture do oraculo
(`tools/make_glm_oracle.py`) provando: (a) o round-trip e' byte-identico
por tensor (mesmo dtype dentro/fora do container), (b) o loader le e valida
o manifesto sem abortar quando consistente, (c) o loader aborta de forma
limpa quando o manifesto e o layout real do disco discordam. A decisao de
QUAIS tensores especificos do GLM-5.2 real precisam ir para int2 para caber
em 8.5 GB, e o logit-diff resultante contra o oraculo em fp32/bf16, ficam
pendentes de acesso ao checkpoint real.

## O que e' real (fixture) vs o que depende de #118 (hardware real de 16 GB)

**Validado nesta issue, com as fixtures disponiveis** (`make_glm_oracle.py`,
`make_glm_bench_model.py` — 313M e tiny, pesos aleatorios, arquitetura real
`glm_moe_dsa`):

- Mecanismo `DUMP_TENSORS=1` emite a tabela tensor-a-tensor correta.
- Container v2 (dtype misto por-tensor): conversor + loader + round-trip
  bit-identico, com teste automatizado.
- Gate duro de RSS: aborta de forma limpa quando a projecao excede o teto
  configurado, com teste automatizado (C real + Python isolado).
- Perfil `16gb` aplica os defaults documentados acima, respeitando
  overrides explicitos do usuario, com teste automatizado.
- O motor carrega e gera texto sob o perfil `16gb` com as fixtures
  disponiveis, sem exceder o budget projetado (adaptado a escala da
  fixture: o teto de 13 GB do perfil real e' MUITO maior do que qualquer
  projecao possivel para um modelo de 313M ou tiny — os testes verificam
  que o gate NAO dispara nesse caso, e disparam corretamente com um teto
  artificialmente pequeno para provar a logica de abort).

**Pendente de #118 (bloqueada por hardware real de 16 GB) ou de acesso ao
checkpoint GLM-5.2 real**:

- RSS de pico MEDIDO (nao projetado) do GLM-5.2 real, em qualquer
  configuracao de quantizacao, numa maquina fisica de 16 GB.
- Confirmacao de que o tronco denso do GLM-5.2 real, com quantizacao mista
  via container v2, efetivamente cabe nos 8.5 GB de sub-orcamento (a
  medicao upstream de ~9.9 GB em int4 uniforme e' documentada mas nao
  remedida aqui).
- Logit-diff do GLM-5.2 real sob o perfil `16gb` contra a referencia em
  bf16/fp32 (tolerancia de correctness da issue).
- Numeros de velocidade (tok/s) reais sob pressao de memoria de 16 GB —
  as fixtures sinteticas rodam rapido demais (poucos MB residentes) para
  serem representativas de latencia de disco sob um cache de experts
  pequeno.

Nao marque estes itens como resolvidos até que #118 seja desbloqueada e a
medicao real seja feita.

## Kimi K2: requisito adicional de disco

O perfil limita RAM, mas não reduz o tamanho persistente do modelo. Para
Kimi K2, planeje aproximadamente 550 GB para o container int4 e mantenha
**ao menos 600 GB de SSD livres** antes de iniciar a conversão. O conversor
é shard-a-shard e resumable, portanto não precisa manter uma segunda cópia
integral do checkpoint; ainda assim, falta medir o tamanho final e o RSS
com o checkpoint publicado numa máquina física de 16 GB.

## Slice local de fixtures para #129: instrumentacao adaptativa

No runtime C++ deste repositorio, os contratos locais agora preservam tambem:

- `learned pin` no pager de experts: o estado passa a diferenciar residents
  comuns de residents promovidos a pin por historico de reuse, com contrato
  de eviccao preferindo experts nao-pinned quando houver escolha.
- `sparsity cache warmup`: uma rota repetida deixa de ser apenas hit/miss e
  passa a marcar a entrada como `warm`, para separar cache frio de cache ja
  estabilizado em fixtures.
- `speculative adaptive lookahead`: o draft/verify comeca conservador
  (`lookahead=1` durante warmup) e so' expande quando a acceptance medida nas
  fixtures se mantem alta; abaixo do threshold, o plano continua
  conservador. Isso preserva losslessness porque o caminho autoritativo e a
  reconciliacao do `PEagleDecoder` continuam sendo a fonte de verdade.

Esses sinais sao **de fixture/contrato local**, nao de hardware real. Os
numeros finais de tok/s, RSS, acceptance quente e beneficio real em maquina
fisica de 16 GB continuam pendentes de #118 e devem ser reportados como
pendentes enquanto nao houver medicao direta.
